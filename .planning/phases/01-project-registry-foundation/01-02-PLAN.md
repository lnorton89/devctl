<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> `/api/projects` | User-controlled project configuration enters backend validation. |
| API -> YAML file | Validated config becomes durable local state. |
| YAML file -> API | Local human-edited YAML re-enters the app and may be malformed or tampered. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-02-01 | Tampering | `registryRepository.ts` | mitigate | Parse YAML then validate with `registrySchema`; reject malformed/invalid data without overwrite. |
| T-01-02-02 | Information Disclosure | `projects.ts` | mitigate | Do not log request bodies, env values, or raw persisted contents; generic error messages only. |
| T-01-02-03 | Denial of Service | `app.ts` | mitigate | Keep JSON body size limit and avoid unbounded parsing paths. |
| T-01-02-04 | Elevation of Privilege | `projects.ts` | mitigate | Store command strings but expose no execution route or lifecycle behavior in Phase 1. |
| T-01-02-05 | Repudiation | `registryRepository.ts` | accept | Phase 1 is trusted single-user local software and does not require an audit log. |
</threat_model>
